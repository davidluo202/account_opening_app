# CMF 開戶系統升級規劃方案

> **文檔目的**：為 CMF 開戶系統的下一階段升級提供全面的規劃建議
> 
> **規劃時間**：2026年5月
> 
> **當前版本**：v1.0（網上開戶提交系統）
> 
> **重要說明**：
> - 公司是香港持牌券商
> - 已有客戶帳戶編碼規則（手工開戶使用中），系統需從指定序列號開始
> - 客戶買賣金融資產會形成持倉，需獨立數據庫
> - 未來支持加密貨幣及資產代幣化（RWA）

## 四、客戶帳戶編號規則（重要）

### 4.1 現有規則

公司已有自己的客戶帳戶編碼規則，在手工開戶中使用。

### 4.2 系統對接要求

**系統投產時需要從指定的序列號開始**，編制後續的客戶帳戶。

### 4.3 配置設計

```typescript
interface AccountNumberConfig {
  // 帳戶號前綴
  prefix: string;           // 如：CMF
  
  // 起始序列號（可配置）
  startSequence: number;    // 如：1000
  
  // 序列號位數
  padding: number;          // 如：4位 → 0001
  
  // 日期格式
  dateFormat: 'YYYYMMDD' | 'YYYYMM' | 'none';
  
  // 帳戶類型標識
  typeIndicator: {
    individual: string;     // 個人：IND
    joint: string;         // 聯名：JNT
    corporate: string;      // 公司：CORP
  };
}
```

### 4.4 生成示例

```
前綴：CMF
類型：IND（個人）
起始：1000
日期：2026-05-07

生成：CMF-IND-20260507-1000
```

### 4.5 序列號管理

```sql
-- 序列號表
CREATE TABLE account_sequences (
  id INT PRIMARY KEY AUTO_INCREMENT,
  accountType VARCHAR(20) NOT NULL,  -- individual/joint/corporate
  lastSequence INT NOT NULL DEFAULT 0,
  updatedAt DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY (accountType)
);
```

## 五、持倉數據庫設計

### 5.1 為什麼需要獨立持倉數據庫

- 客戶持倉不僅是價值，更是具體的資產數量
- 不同產品有不同的屬性和結算方式
- 需要追蹤每筆資產的成本基礎

### 5.2 持倉表結構

```typescript
interface Position {
  id: string;
  accountNumber: string;   // 關聯客戶帳戶
  
  // 資產標識
  assetCode: string;         // 如：0700.HK（騰訊）
  assetName: string;
  assetType: AssetType;
  
  // 數量
  quantity: number;          // 如：500股
  avgCost: number;           // 平均成本
  
  // 市值
  currentPrice: number;
  marketValue: number;
  
  // 盈虧
  unrealizedPL: number;
  realizedPL: number;
  
  openDate: Date;
}

type AssetType = 
  | 'stock'      // 股票
  | 'warrant'   // 窩輪
  | 'bond'      // 債券
  | 'etf'       // ETF
  | 'futures'   // 期貨
  | 'options'   // 期權
  | 'crypto'    // 加密貨幣（未來）
  | 'rwa';      // 代幣化資產（未來）
```
